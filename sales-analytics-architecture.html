01
Kaggle → Local
Brazilian E-Commerce Dataset (Olist)
9 CSV files · 99,441 orders · customer + product + seller data
FREE

Source

Olist Brazilian e-commerce public dataset on Kaggle. Real marketplace data from 2016–2018. 9 relational CSV files.

Schema

Orders, customers, products, sellers, reviews, payments, geolocation. Star schema — orders as fact table, others as dimensions.

Scale

99,441 orders, 96,096 customers, 32,951 products, 3,095 sellers. Rich data for analytics — revenue, CLV, churn, segmentation.

"Real marketplace data from a Brazilian e-commerce platform — not synthetic. 9 relational tables gives us a production-like schema for complex joins."
↓
02
Apache Spark (PySpark 3.5) · Google Colab
PySpark ETL — Distributed Processing
Read CSVs → Join 9 tables → Window functions → Aggregations → Write PostgreSQL
FREE

Spark Session

PySpark 3.5 on Google Colab. SparkSession with jdbc connector to PostgreSQL. 4 partitions by customer_id.

Window Functions

Running totals per customer using Window.partitionBy("customer_id").orderBy("order_date"). Customer lifetime value calculation.

Optimizations

Broadcast join for small dimension tables (<10MB). In-memory caching of frequently accessed DataFrames. Repartitioning by customer_id.

Performance

Processing time reduced from 45s → 14.3s (68% improvement) through caching + partitioning + broadcast variables.

"Broadcast joins for small dimension tables (products, sellers) eliminated shuffle operations — the biggest performance win from 45s to 14.3s."
↓
03
PostgreSQL 15 · Docker
PostgreSQL — Analytical Data Store
Indexes + materialized views + optimized queries → 60% faster analytics
LOCAL

Schema Design

Normalized schema with orders, customers, products, order_items tables. Foreign key constraints for data integrity.

Indexes

Composite index on (customer_id, order_date). Partial index on active orders. B-tree indexes on all JOIN columns.

Materialized Views

mv_customer_stats — pre-aggregated CLV and order counts. Refreshed daily. 60% query speedup vs live aggregation.

"Materialized views pre-compute expensive aggregations — customer lifetime value query went from 2.3s to 180ms. Worth the storage cost."
↓
04
Spring Boot 3.5 · Java 17
REST API — 7 Analytics Endpoints
SOLID principles · Hibernate/JPA · Resilience4j · <100ms p99 latency
LOCAL

7 Endpoints

GET /health · GET /orders · POST /orders · GET /orders/{id} · GET /analytics/summary · GET /analytics/top-customers · GET /analytics/status

Architecture

Controller → Service → Repository layers. SOLID principles throughout. Custom exception handling with proper HTTP status codes.

Performance

Hibernate 2nd level cache for frequently-read data. Native SQL for complex analytical queries. Connection pooling with HikariCP.

"Service layer pattern with dependency injection means we can swap PostgreSQL for any database — zero changes to controller or business logic."

05
Docker Compose
Containerized Deployment
PostgreSQL + Spring Boot in Docker Compose · Health checks · Persistent volumes
LOCAL

Services

2 containers: postgres:15 on port 5432, sales-api Spring Boot on port 8080. Depends-on with health checks.

Volumes

Persistent volume for PostgreSQL data — survives container restarts. Environment variables for DB credentials.

Swagger

SpringDoc OpenAPI 3.0 auto-generates Swagger UI at localhost:8080/swagger-ui.html. All 7 endpoints documented.

"Docker Compose means one command deploys both database and API — reproducible across any machine with no environment setup."
