End-to-End Architecture — Click Each Stage
01
NYC TLC Website → Local
Download NYC TLC Parquet Files
Jan–Mar 2024 trip records · 160MB Parquet · Official government data
FREE

Source

NYC Taxi & Limousine Commission publishes monthly trip records at nyc.gov/tlc. Official government open data — no API key needed.

Format

Pre-partitioned Parquet files by month. Jan–Mar 2024 = 9.2M trip records. Columns: pickup/dropoff times, locations, fares, payment types.

Scale

160MB compressed Parquet. Expands to ~600MB in memory. Efficient columnar format perfect for analytical workloads.

"We used the official NYC TLC dataset — 9.2M trips from Jan–Mar 2024. Government open data means reproducible results anyone can verify."
↓
02
Docker Compose → Kafka + Zookeeper
Kafka Producer — Real-Time Streaming Ingestion
Python producer reads Parquet → publishes to Kafka topic at 57 msgs/sec
LOCAL

Producer

Python Kafka producer reads trip records from Parquet file and publishes each row as a JSON message to nyc-taxi-trips topic.

Throughput

57 messages/sec sustained throughput. Simulates real-time taxi dispatch system receiving live trip completions.

Docker Setup

Kafka + Zookeeper + Kafka UI all running in Docker Compose. kafka:9092 internal, localhost:9092 external. Persistent volumes.

"Kafka decouples the data source from consumers — adding a new consumer (say, a real-time fraud detector) requires zero changes to the producer."
↓
03
Kafka Consumer → DuckDB
Kafka Consumer — Stream Processing + Ingestion
Python consumer reads topic → validates → inserts into DuckDB analytical warehouse
LOCAL

Consumer Group

Python consumer with group_id=nyc-taxi-consumer. Auto-offset reset to earliest. Batch inserts every 1000 messages for efficiency.

Data Validation

Schema validation on each message — checks fare > 0, trip distance > 0, valid payment type, valid location IDs. Bad records logged and skipped.

DuckDB Insert

Batch inserts into DuckDB trips table. DuckDB's COPY command processes 1000-row batches in microseconds.

"Batch inserts of 1000 records at a time instead of row-by-row gave us 50x better ingestion throughput into DuckDB."
↓
04
Apache Airflow DAG
Airflow Orchestration — Daily ETL Pipeline
DAG scheduled daily @ 6 AM UTC · 5 tasks · 99%+ success rate
LOCAL

DAG Schedule

Cron 0 6 * * * — runs daily at 6 AM UTC. Retries=2 with 5-minute delay. Email alerts on failure.

5 Tasks

validate_source → extract_parquet → transform_clean → load_duckdb → trigger_dashboard. Each task isolated with XCom passing.

Monitoring

Airflow UI at localhost:8080. Task durations, logs, retry history all visible. SLA misses trigger email notification.

"Airflow's task dependency graph means if the extract fails, transform never runs — preventing partial loads that corrupt analytics."
↓
05
DuckDB — OLAP Engine
Analytical Queries — Sub-100ms Performance
Revenue analysis · Hourly patterns · Location analytics · Payment distribution
FREE

Why DuckDB

In-process OLAP database — no server needed. Columnar storage with vectorized execution. Sub-100ms on 9.2M rows on a laptop.

Key Queries

Total revenue $253M, peak hour (6 PM = 666K trips), top location (JFK = 800K trips), payment split (78% credit card).

vs Alternatives

DuckDB vs PostgreSQL: 10x faster analytical queries. DuckDB vs Spark: no cluster needed for 160MB data. Right tool for right scale.

"DuckDB was the right choice for this scale — sub-100ms analytical queries on 9.2M records with zero infrastructure. Spark would be overkill for 160MB."
